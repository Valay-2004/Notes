# Java Generics

Generics were introduced to provide **compile-time type safety** and eliminate the need for explicit type casting.

#### The Evolution of Collections

Before Generics, collections stored objects as the `Object` type, leading to potential `ClassCastException` at runtime.

```java
// The Old Way (Pre-Java 5)
List myList = new ArrayList();
myList.add("Hello");
myList.add(123); // Perfectly legal, but dangerous!

String s = (String) myList.get(0); // Explicit cast required
String s2 = (String) myList.get(1); // Runtime crash!
```

#### Modern Approach with Generics

```java
List<String> myList = new ArrayList<>();
myList.add("Hello");
// myList.add(123); // Compile-time error! Type safety ensured.
String s = myList.get(0); // No casting needed.
```

---

> [!abstract] What are Generics?
> Generics let you write **reusable code** that works with **any data type** while keeping **type safety** at compile time.  
> Instead of writing separate classes/methods for int, String, Dog, etc., you write **one** using a "placeholder" (type parameter like <T>).

> Analogy: Think of a **reusable plastic box** (generic class). You decide what goes inside: fruits, toys, books — but once decided, only that type is allowed (no mixing).  
> Without generics → use Object (loses type safety, needs casting, runtime errors possible).  
> With generics → compiler checks types early → no ClassCastException surprises!

**Why Generics?**

- **Type Safety** — Catch wrong types at **compile time** (not runtime crash).
- **No Casting Needed** — Cleaner code: `String s = list.get(0);` instead of `(String) list.get(0)`.
- **Code Reusability** — One class/method for many types.
- **Better Readability** — `List<String>` tells exactly what's inside.
- **Used Everywhere** — Collections (ArrayList<String>), custom data structures, etc.

**Before Generics (Java 1.4 style – Problem Demo)**

```java
import java.util.ArrayList;

public class BeforeGenerics {
    public static void main(String[] args) {
        ArrayList list = new ArrayList();   // Raw type – no < >
        list.add("Hello");
        list.add(123);                      // Allowed! Mix types
        list.add(new Dog());                // Even objects

        String s = (String) list.get(0);    // Need casting
        // String num = (String) list.get(1); // Runtime ClassCastException!
    }
}
```

→ Dangerous: compiles fine, crashes at runtime.

## 1. Generic Classes

Create a class with type parameter <T> (T = Type, can use any letter: E, K, V, etc.).

```java
// Generic Class Example
class Box<T> {                  // T is placeholder
    private T item;             // Can hold any type

    public void setItem(T item) {
        this.item = item;
    }

    public T getItem() {
        return item;
    }
}

public class GenericClassDemo {
    public static void main(String[] args) {
        Box<String> stringBox = new Box<>();    // Diamond operator (<>) infers type
        stringBox.setItem("Apple");
        String fruit = stringBox.getItem();     // No casting!
        System.out.println(fruit);

        Box<Integer> intBox = new Box<>();
        intBox.setItem(100);
        // intBox.setItem("Hello");  // Compile error! Type safety

        Box<Dog> dogBox = new Box<>();          // Works with custom class too
    }
}
```

**Deep Points:**

- Multiple parameters: `class Pair<K, V> { private K key; private V value; }` (like Map.Entry).
- Generic class can extend non-generic or another generic.

## 2. Generic Methods

Methods can be generic (even in non-generic classes). Type parameter before return type.

```java
class Utils {
    // Generic method <T>
    public static <T> void print(T item) {
        System.out.println("Item: " + item);
    }

    // With return type
    public static <T> T getFirst(T[] array) {
        return array[0];
    }
}

public class GenericMethodDemo {
    public static void main(String[] args) {
        Utils.print("Hello");       // T inferred as String
        Utils.print(42);            // T inferred as Integer

        String[] names = {"A", "B"};
        String first = Utils.getFirst(names);   // No cast
    }
}
```

**When to use?** Utility methods that work on any type.

## 3. Bounded Type Parameters (<T extends ...>)

Restrict T to specific types (upper bound).

```java
// T must be Number or subclass (Integer, Double, etc.)
class Calculator<T extends Number> {
    private T value;

    public Calculator(T value) {
        this.value = value;
    }

    public double doubleValue() {
        return value.doubleValue();   // Safe because Number has doubleValue()
    }
}

// Multiple bounds: <T extends Number & Comparable<T>>
```

**Lower Bound?** Not directly on type param — use wildcards for that.

## 4. Wildcards (?) – The Flexible Part

Wildcard = unknown type. Used in parameters for flexibility.

| Wildcard Type | Syntax             | Meaning / Use Case                                 | Can Add Elements?  |
| ------------- | ------------------ | -------------------------------------------------- | ------------------ |
| Unbounded     | `?`                | Any type – read-only (get as Object)               | No                 |
| Upper Bounded | `? extends Number` | Number or subclass – read safely as Number         | No (compiler safe) |
| Lower Bounded | `? super Integer`  | Integer or superclass – write safely (add Integer) | Yes                |

**PECS Rule** (Producer Extends, Consumer Super) – Memory Trick:

- If method **reads** from collection → use `extends` (producer).
- If method **writes** to collection → use `super` (consumer).

```java
import java.util.ArrayList;
import java.util.List;

class WildcardDemo {
    // Upper bound: read as Number
    static void printNumbers(List<? extends Number> list) {
        for (Number n : list) {
            System.out.println(n);
        }
        // list.add(10);  // ERROR! Cannot add – don't know exact type
    }

    // Lower bound: write Integer or subclass
    static void addIntegers(List<? super Integer> list) {
        list.add(100);          // Safe
        list.add(new Integer(200));
        // Object obj = list.get(0);  // Only Object guaranteed
    }

    public static void main(String[] args) {
        List<Integer> ints = new ArrayList<>();
        ints.add(5);
        printNumbers(ints);     // Works (Integer extends Number)

        List<Number> nums = new ArrayList<>();
        addIntegers(nums);      // Works
    }
}
```

## 5. Type Erasure (What Happens Behind the Scenes)

Generics are compile-time only → JVM sees raw types at runtime (erasure).

- `List<String>` → becomes `List` at runtime.
- No runtime type checks for generics → that's why no `new T()` or `T[] array = new T[10];` (use Array.newInstance or Object[] + cast).
- Benefits: Backward compatibility with old code.

**No:**

```java
T var = new T();          // Error – erased
if (obj instanceof List<String>) {}  // Error – erased
```

## Quick Summary Table

| Feature            | Purpose / Syntax       | Key Rule / Limitation             |
| ------------------ | ---------------------- | --------------------------------- |
| Generic Class      | class Box<T> {}        | Type param for fields/methods     |
| Generic Method     | <T> void print(T t) {} | Can be in non-generic class       |
| Bounded Type       | <T extends Number>     | Restrict to subclass + interfaces |
| Unbounded Wildcard | List<?>                | Read as Object, no add            |
| Upper Wildcard     | List<? extends Animal> | Read as Animal, no add            |
| Lower Wildcard     | List<? super Dog>      | Add Dog, read as Object           |
| Type Erasure       | Compiler removes <T>   | No runtime generics info          |

> [!success] Memory Trick
>
> - Generics = "Smart Box with Lock" → Decide type once (at creation), compiler locks it forever.
> - Wildcards = "Flexible Door" → ? extends = read-only door, ? super = write-friendly door.
> - Raw types = Old broken lock → Avoid! (Compiler warns).

> [!warning] Common Mistakes
>
> - Using raw types: `List list = new ArrayList();` → loses safety.
> - Adding wrong type → compile error (good!).
> - Creating arrays of generics: `List<String>[] arrays;` allowed but tricky.
> - Forgetting diamond <> in Java 7+.

## 🙋 Interview Corner

**Q1: What is Type Erasure?**
A: It is the process where the compiler removes all generic type information at compile-time and replaces it with the raw type (usually `Object`). This ensures backward compatibility with older Java versions.

**Q2: What is the "PECS" principle?**
A: **Producer Extends, Consumer Super.**

- If you only _get_ items from a collection, use `<? extends T>`.
- If you only _add_ items to a collection, use `<? super T>`.

**Q3: Why can't we use primitives (like `int`) in Generics?**
A: Generics work by treating types as `Object`. Since primitives do not inherit from `Object`, you must use their **Wrapper classes** (e.g., `Integer`, `Double`).

**Q4: Can we create an array of a generic type?**
A: **No.** You cannot do `T[] arr = new T[10];` because the exact type of `T` is unknown at runtime due to Type Erasure. You must use `(T[]) new Object[10]` or `Array.newInstance()`.

**Q5: What is a Raw Type and why should we avoid it?**
A: A raw type is using a generic class without a type parameter (e.g., `List` instead of `List<String>`). It should be avoided because it bypasses compile-time type checks and leads to unsafe code.
