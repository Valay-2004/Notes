# Encapsulation in Java - Quick Revision Notes

> [!info] Quick Navigation
> Next: [[02. Inheritance.md|Inheritance]] | Related: [[03. Abstraction.md|Abstraction]], [[05. Final, Super & Access Modifiers in JAVA.md|Access Modifiers]]


> [!abstract] Core Idea
> Hide internal details of an object → expose only necessary parts  
> → Bundle (encapsulate) data + methods together  
> → One of the 4 pillars of OOP

> [!question] Why Use Encapsulation?
> - Protect data from invalid/unwanted changes  
> - Add validation/control when modifying data  
> - Easier to change internal implementation later (without breaking code that uses the class)  
> - Improves **security**, **maintainability**, and **flexibility**

## How to Achieve It (The Standard Way)

> [!tip] Two Simple Steps
> 1. Declare variables as **`private`**  
> 2. Provide **`public`** getter & setter methods

```java
class Student {
    private String name;   // hidden from outside
    private int age;

    // Getter (read access)
    public String getName() {
        return name;
    }

    // Setter (controlled write access + validation)
    public void setName(String name) {
        if (name != null && !name.trim().isEmpty()) {
            this.name = name;
        } else {
            System.out.println("Invalid name!");
        }
    }

    public int getAge() { return age; }

    public void setAge(int age) {
        if (age > 0 && age < 150) {
            this.age = age;
        }
    }
}
```

> [!important] Key Rules to Remember
> - **Private** variables → accessible only inside the class  
> - **Getters** → usually just `return` the value  
> - **Setters** → use `this.` to distinguish field from parameter + add checks  
> - No public fields! → that's the opposite of encapsulation

> [!warning] Common Mistake (Avoid This!)
> ```java
> // BAD - No protection
> public int age;   // anyone can do: obj.age = -999;
> ```
> 
> ```java
> // GOOD - Safe & controlled
> private int age;
> public void setAge(int age) { if (age > 0) this.age = age; }
> ```

## Benefits at a Glance

> [!success] Quick Wins
> - **Data security** — reject bad values (age = -5? Nope!)  
> - **Flexibility** — change how age is stored (e.g., switch to LocalDate) without affecting users  
> - **Better debugging** — logic is in one place (setters)  
> - Supports future OOP concepts like abstraction

## Memory Trick / One-Liner

> [!tip] Capsule Analogy (Navin-Style)
> Think of a **medicine capsule** → medicine (data) is hidden inside → you only interact via the outer shell (public methods).  
> Encapsulation = **Private data** + **Public controlled access** = Safe & Clean Code!

## Quick Comparison Table

| Aspect               | Without Encapsulation (public fields) | With Encapsulation (private + get/set) |
|----------------------|---------------------------------------|----------------------------------------|
| Data Protection      | None                                 | High (validation possible)             |
| Direct Modification  | Yes (dangerous)                      | No — only through methods              |
| Future Changes       | Breaks outside code                  | Internal changes invisible             |
| Validation           | Impossible                           | Easy in setters                        |
| OOP Principle        | Violates                             | Follows perfectly                      |
