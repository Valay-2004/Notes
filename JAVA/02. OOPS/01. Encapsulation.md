# Encapsulation in Java

#### TL;DR (Quick Revision)

**Encapsulation** = Hide internal details (private data) + Provide controlled access (public methods)

> [!abstract] Core Idea
> Bundle data + methods together → Hide internal details → Expose only necessary parts through public getters/setters → One of the 4 pillars of OOP

> [!question] Why Use It?
>
> - Protect data from invalid changes via validation
> - Change internal implementation without breaking external code
> - Control access and improve security/maintainability
> - Enable future OOP features (abstraction, composition)

#### Quick Comparison Table

| Aspect              | Without Encapsulation (public fields) | With Encapsulation (private + get/set) |
| ------------------- | ------------------------------------- | -------------------------------------- |
| Data Protection     | None — anyone can assign any value    | High — validation in setters           |
| Direct Modification | Yes (dangerous)                       | No — only through controlled methods   |
| Future Changes      | Breaks outside code                   | Internal changes invisible to users    |
| Validation          | Impossible                            | Easy in setters                        |
| OOP Principle       | Violates                              | Follows perfectly                      |

## The Learning Concept

### What Problem Does It Solve?

Imagine a `BankAccount` class without encapsulation:

```java
// BAD - No protection
class BankAccount {
    public double balance;  // Anyone can modify directly!
}

BankAccount acc = new BankAccount();
acc.balance = 1000;
acc.balance = -5000;  // What?! Negative balance? Account destroyed!
```

This is dangerous because there's **no validation**. Encapsulation fixes this:

```java
// GOOD - Safe & controlled
class BankAccount {
    private double balance;  // Hidden from outside

    // Only way to modify: through a validated setter
    public void setBalance(double amount) {
        if (amount >= 0) {
            this.balance = amount;  // Only valid values accepted
        } else {
            System.out.println("Error: Balance cannot be negative!");
        }
    }

    public double getBalance() {
        return balance;
    }
}
```

### How Java Implements It

> [!tip] Two Simple Steps
>
> 1. Declare variables as **`private`** (hidden from outside)
> 2. Provide **`public`** getter & setter methods (controlled access)

#### Full Runnable Example

```java
class Student {
    // Step 1: Private variables (hidden)
    private String name;
    private int age;
    private double gpa;

    // Step 2: Public getters (read access)
    public String getName() {
        return name;
    }

    public int getAge() {
        return age;
    }

    public double getGpa() {
        return gpa;
    }

    // Step 2: Public setters (controlled write access with validation)
    public void setName(String name) {
        if (name != null && !name.trim().isEmpty()) {
            this.name = name;  // Valid → accept
            System.out.println("✓ Name set to: " + name);
        } else {
            System.out.println("✗ Invalid name! Cannot be empty.");
        }
    }

    public void setAge(int age) {
        if (age > 0 && age < 150) {
            this.age = age;  // Valid → accept
            System.out.println("✓ Age set to: " + age);
        } else {
            System.out.println("✗ Invalid age! Must be between 1-149.");
        }
    }

    public void setGpa(double gpa) {
        if (gpa >= 0.0 && gpa <= 4.0) {
            this.gpa = gpa;  // Valid → accept
            System.out.println("✓ GPA set to: " + gpa);
        } else {
            System.out.println("✗ Invalid GPA! Must be between 0.0-4.0.");
        }
    }
}

// Main method to demonstrate
public class Main {
    public static void main(String[] args) {
        Student student = new Student();

        // Using setters (validation happens automatically)
        student.setName("Alice");           // ✓ Valid
        student.setName("");                // ✗ Invalid

        student.setAge(20);                 // ✓ Valid
        student.setAge(-5);                 // ✗ Invalid

        student.setGpa(3.8);                // ✓ Valid
        student.setGpa(5.0);                // ✗ Invalid

        // Using getters (read-only access)
        System.out.println("\nFinal Values:");
        System.out.println("Name: " + student.getName());
        System.out.println("Age: " + student.getAge());
        System.out.println("GPA: " + student.getGpa());
    }
}
```

**Output:**

```
✓ Name set to: Alice
✗ Invalid name! Cannot be empty.
✓ Age set to: 20
✗ Invalid age! Must be between 1-149.
✓ GPA set to: 3.8
✗ Invalid GPA! Must be between 0.0-4.0.

Final Values:
Name: Alice
Age: 20
GPA: 0.0
```

### Key Terminology

1. **Getter**: A public method that returns the value of a private field.
   Syntax: `public TypeName getFieldName() { return fieldName; }`

2. **Setter**: A public method that safely modifies a private field. Includes validation logic.
   Syntax: `public void setFieldName(TypeName value) { ...validation... this.fieldName = value; }`

3. **Validation**: Checks performed in a setter to ensure only valid data is assigned.
   Example: `if (age > 0 && age < 150)`

4. **`this` Keyword**: Refers to the current object. Used to distinguish instance variables from method parameters.
   Example: `this.age = age;` means "assign the parameter `age` to the field `this.age`"

> [!important] Key Rules to Remember
>
> - **Private** variables → accessible **only inside the class**
> - **Getters** → usually just `return` the value (no validation needed)
> - **Setters** → add validation **before** assignment
> - **Use `this.`** to distinguish field from parameter when names are the same
> - **Never make fields public** → breaks encapsulation

## ⚡ Edge Case Analysis

When implementing encapsulation, consider these tricky scenarios:

### 1. Handling Null in Setters

**Problem:** Passing `null` to a setter for a `String` or `Object` can lead to `NullPointerException` later.
**Fix:** Always check for null before assigning.

```java
public void setName(String name) {
    if (name == null) {
        throw new IllegalArgumentException("Name cannot be null");
    }
    this.name = name;
}
```

### 2. The "Escaping" Reference (Mutable Objects)

**Problem:** If your private field is a `List` or `Date`, returning it directly via a getter allows anyone to modify your "private" data.
**Fix:** Return a **Defensive Copy** (as shown in the "Going Deeper" section).

### 3. Setter Validation for Numbers

**Problem:** Negative ages, zero prices, or overflow values.
**Fix:** Use range-based validation.

```java
public void setPrice(double price) {
    if (price < 0) {
        System.out.println("Error: Price cannot be negative");
        return;
    }
    this.price = price;
}
```

## Going Deeper (Intermediate → Advanced)

### Defensive Copying (Mutable Objects)

If a field is a **mutable object** (like `List`, `Date`, `Array`), returning it directly from a getter **breaks encapsulation**:

```java
// DANGER: Breaks encapsulation
class Team {
    private List<String> members = new ArrayList<>();

    public List<String> getMembers() {
        return members;  // Oops! External code can modify the internal list
    }
}

Team team = new Team();
team.getMembers().add("Hacker");  // Modified internal state directly!
```

**Solution: Defensive Copy**

```java
// SAFE: Encapsulation preserved
import java.util.Collections;

class Team {
    private List<String> members = new ArrayList<>();

    public List<String> getMembers() {
        return Collections.unmodifiableList(members);  // Return immutable copy
    }

    // Or create a new list:
    // return new ArrayList<>(members);
}

Team team = new Team();
team.getMembers().add("Hacker");  // ERROR! Can't modify unmodifiable list
```

### Immutability (The Strongest Encapsulation)

An **immutable class** has no setters — once an object is created, it cannot be modified:

```java
final class ImmutableStudent {  // final = cannot be extended
    private final String name;
    private final int age;

    // Constructor (only way to set values)
    public ImmutableStudent(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // Only getters (no setters!)
    public String getName() { return name; }
    public int getAge() { return age; }
}

ImmutableStudent student = new ImmutableStudent("Alice", 20);
// student.name = "Bob";  // ERROR! Field is final (cannot reassign)
// student = new ImmutableStudent("Bob", 25);  // This is allowed, but creates a new object
```

**Advantages:**

- ✅ Thread-safe (no synchronization needed)
- ✅ Can be used as HashMap keys
- ✅ Easier to reason about (state never changes)

### Java Records (Java 14+) - Modern Encapsulation

Java Records automatically generate getters, equals(), hashCode(), toString():

```java
// Old way (verbose)
class Student {
    private String name;
    private int age;

    public Student(String name, int age) { this.name = name; this.age = age; }
    public String getName() { return name; }
    public int getAge() { return age; }
    // ... equals(), hashCode(), toString() ...
}

// New way (concise) - Java 14+
record Student(String name, int age) { }

// Usage is identical
Student s = new Student("Alice", 20);
System.out.println(s.name());  // Alice
System.out.println(s.age());   // 20
```

> [!warning] Records are immutable by default
> Fields in records are `final` and there are no setters. Perfect for data classes!

### Access Modifiers Beyond private/public

| Modifier               | Inside Class | Same Package | Subclass | Outside |
| ---------------------- | :----------: | :----------: | :------: | :-----: |
| `private`              |      ✅      |      ❌      |    ❌    |   ❌    |
| package-private (none) |      ✅      |      ✅      |    ❌    |   ❌    |
| `protected`            |      ✅      |      ✅      |    ✅    |   ❌    |
| `public`               |      ✅      |      ✅      |    ✅    |   ✅    |

**When to use:**

- `private` → Default for fields; only expose what's needed
- `protected` → For subclasses in inheritance hierarchies
- `public` → For getters/setters and public APIs

### Encapsulation vs Abstraction (Common Confusion)

| Feature        | Encapsulation                                         | Abstraction                                    |
| -------------- | ----------------------------------------------------- | ---------------------------------------------- |
| **What it is** | Hiding _implementation details_ (data + methods)      | Hiding _complexity_ behind a simple interface  |
| **How**        | `private` fields + `public` getters/setters           | Interfaces, abstract classes, methods          |
| **Example**    | `private` balance field with validated `setBalance()` | `move()` method that works for Car, Bike, Boat |
| **Goal**       | Data protection & control                             | Simplify usage & reduce complexity             |

> [!tip] Simple Way to Remember
>
> - **Encapsulation** = HOW you hide things (private/public)
> - **Abstraction** = WHAT you hide (complex details)

### Lombok: Reducing Boilerplate

In practice, nobody writes getters/setters by hand. **Lombok** generates them automatically:

```java
import lombok.Getter;
import lombok.Setter;

@Getter
@Setter
class Student {
    private String name;
    private int age;
}

// Lombok generates getName(), setName(), getAge(), setAge() automatically!
```

Alternative: **IDE generation** — most IDEs can generate getters/setters via menu → right-click → Generate

## Interview Corner

### Common Questions & Concise Answers

**Q1: What is encapsulation?**  
A: Bundling data (fields) + methods together and hiding internal details behind a public interface. Achieved using `private` fields + `public` getters/setters.

**Q2: What's the difference between encapsulation and abstraction?**  
A: Encapsulation hides _how_ things work (implementation). Abstraction hides _what_ things are (complexity). Encapsulation is a _technique_, abstraction is a _concept_.

**Q3: Why use private fields instead of public?**  
A: Public fields have no protection — anyone can assign invalid data. Private fields force validation through setters, ensuring data integrity.

**Q4: What's defensive copying? When do you need it?**  
A: Returning an immutable copy or unmodifiable view instead of the actual mutable object. Needed when fields are mutable objects (List, Date, Array).

**Q5: What's the benefit of immutable classes?**  
A: They're thread-safe, hashable, and easier to reason about. Once created, their state never changes.

**Q6: How are Java Records different from regular classes?**  
A: Records automatically generate getters, equals(), hashCode(), toString(). They're immutable by default (all fields are final, no setters).

### Classic Mistakes / Traps

❌ **Mistake 1:** Making fields public for "convenience"

```java
public String name;  // DON'T! No validation possible
```

✅ **Fix:** Always use `private` by default.

❌ **Mistake 2:** Returning mutable objects directly from getters

```java
public List<String> getItems() { return items; }  // User can modify!
```

✅ **Fix:** Return unmodifiable or defensive copy:

```java
public List<String> getItems() { return Collections.unmodifiableList(items); }
```

❌ **Mistake 3:** Forgetting `this` when parameter name matches field name

```java
public void setAge(int age) {
    age = age;  // This assigns age to itself (useless!)
}
```

✅ **Fix:** Use `this`:

```java
public void setAge(int age) {
    this.age = age;  // Correct: assigns parameter to field
}
```

❌ **Mistake 4:** Setters with no validation

```java
public void setAge(int age) {
    this.age = age;  // No checks → could be -5!
}
```

✅ **Fix:** Always validate:

```java
public void setAge(int age) {
    if (age > 0 && age < 150) {
        this.age = age;
    }
}
```
