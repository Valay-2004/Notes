# Abstraction in Java

## Core Idea

- Hide **implementation details** → show only **essential features** / functionality to the user.
- → "Show what it does, not how it does it"
- → One of the 4 pillars of OOP (with Encapsulation, Inheritance, Polymorphism)

## Two Ways to Achieve Abstraction

### 1. Abstract Class (Partial Abstraction – 0% to 100%)

- Use keyword **`abstract`**
- Can have both **abstract methods** (no body) + **concrete methods** (with body)
- Cannot create object: `new AbstractClass()` → ❌
- Child class **must** implement all abstract methods (or become abstract itself)

```java
// Abstract class example
abstract class Vehicle {                // incomplete class
    abstract void start();              // no body → must override

    void stop() {                       // concrete method (optional)
        System.out.println("Vehicle stopped.");
    }
}

class Bike extends Vehicle {
    @Override
    void start() {
        System.out.println("Bike started with kick.");
    }
}

public class Main {
    public static void main(String[] args) {
        Vehicle v = new Bike();         // Upcasting – allowed
        v.start();                      // Bike started with kick.
        v.stop();                       // Vehicle stopped.
        // Vehicle v2 = new Vehicle();  // ERROR – cannot instantiate
    }
}
```

### 2. Interface (100% Abstraction – before Java 8)

- Use keyword **`interface`**
- All methods are **public abstract** by default (no need to write abstract)
- No concrete methods (until Java 8 – default & static methods added later)
- Class can **`implements`** multiple interfaces → solves multiple inheritance issue
- 100% abstraction (pure contract – no implementation)

```java
// Interface – 100% abstraction
interface Animal {
    void eat();                         // public abstract by default
    void sleep();
}

class Dog implements Animal {
    @Override
    public void eat() {
        System.out.println("Dog eats bones.");
    }

    @Override
    public void sleep() {
        System.out.println("Dog sleeps.");
    }
}

public class Main {
    public static void main(String[] args) {
        Animal a = new Dog();           // reference of interface
        a.eat();                        // Dog eats bones.
        a.sleep();                      // Dog sleeps.
    }
}
```

> [!important] Key Differences – Quick Table

| Feature           | Abstract Class                | Interface (pre-Java 8)                 |
| ----------------- | ----------------------------- | -------------------------------------- |
| Keyword           | `abstract class`              | `interface`                            |
| Methods           | Abstract + Concrete           | Only abstract (100%)                   |
| Variables         | Can have instance/static vars | Only `public static final` (constants) |
| Inheritance       | `extends` (single)            | `implements` (multiple allowed)        |
| Instantiation     | Cannot create object          | Cannot create object                   |
| Abstraction Level | Partial (0–100%)              | 100%                                   |
| Use when...       | Common code/behavior to share | Pure contract / multiple inheritance   |

> [!example] Real-world Analogy
> Remote control of TV → you press "Power", "Volume" (functionality shown)  
> You don't know internal circuits/wires (implementation hidden)  
> → That's **Abstraction**!

> [!success] Memory Trick
>
> - Abstract class = **partial blueprint** (some ready parts + some to-dos)
> - Interface = **full contract** (only to-dos, no ready parts) → 100% abstraction  
>   → "Interface = 100% promise, no delivery yet"

## ⚡ Edge Case Analysis

### 1. Abstract Class with No Abstract Methods

**Scenario:** Creating an abstract class that only has concrete methods.
**Purpose:** This prevents the class from being instantiated directly while still providing full implementation to its children. Useful for utility base classes.

### 2. Can an Interface have Variables?

**Scenario:** Defining `int x = 10;` in an interface.
**Fact:** Yes, but they are implicitly `public static final` (constants). You cannot have instance variables in an interface.

### 3. Abstract Class Constructor

**Scenario:** Does `abstract class Animal { Animal() { ... } }` make sense?
**Fact:** Yes! Even though you can't do `new Animal()`, the constructor is called during **Constructor Chaining** when a child object (like `new Dog()`) is created. It's used to initialize common fields.

## Summary

- Hiding the implementation of the code and showing a user-friendly version to the user.
- Achieved by **Abstract Classes**
- Through **Interfaces** we can achieve **100% abstraction**.

## 🙋 Interview Corner

**Q1: Can we make an abstract method `static`?**
A: **No.** Static methods belong to the class and cannot be overridden. Since abstract methods _must_ be overridden to be useful, making them static is a compile-time error.

**Q2: Can we make an abstract method `final`?**
A: **No.** `final` prevents overriding, but `abstract` requires it. They are contradictory.

**Q3: Interface vs Abstract Class (Java 8+)?**

- **Abstract Class:** Can have state (instance variables), constructors, and partial implementation. Use for "is-a" relationships.
- **Interface:** Cannot have instance variables or constructors. Since Java 8, can have `default` and `static` methods. Use for "can-do" behaviors (contracts).

**Q4: What is 100% Abstraction?**
A: It's a concept where a class or interface provides only the specification without any implementation. Interfaces were the only way to achieve this before Java 8.

**Q5: Can an abstract class implement an interface?**
A: **Yes.** It doesn't even have to implement the interface methods; it can leave them for the first concrete child class to implement.
