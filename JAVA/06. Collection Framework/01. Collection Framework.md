# Collection Framework: The Foundation

> [!NOTE] What is a collection ?
> A `collection` is simply a group of objects, materials, or any other things that are in a group.

So, Similar to that `Collection Framework` can be said to be a group of `set of Classes` and `set of Interfaces`.

### BUT WHY DO WE NEED THIS ?

We know `Primitive Data Types` are used to store single value into a variable
For example:

- `int x = 10;`
- `int y = 25;`
- `int xy = 30;`

So if one wants to store 1000s of values, then we would need to declare that many number of variables in our program which is not feasible at all.

> I mean writing a thousand values and using them would be like a LOT OF WORK.

Thus, Here comes the concept of `Arrays` in which we can store many elements having same data type.

> [!abstract] Definition
> An `array` is a `data structure` that stores a fixed-size, sequential collection of elements of the same data type in contiguous memory locations.

The main advantage of is that we can store a huge number of elements with a single variable like:

```java
int[] arr = new int[5];
arr[0] = 1;
...
arr[4] = 5;
```

In the above code we've created an array with a length of `5` then we have initialized indexes with elements form 1 to 5.
([Arrays are 0 indexed](https://albertkoz.com/why-does-array-start-with-index-0-65ffc07cbce8))

### **Limitations of Arrays**

> [!warning] Fixed Size
> As Arrays are fixed in size, we cannot `increase` or `decrease` the size of an array based on our requirements. To use an array, we must have a fixed size in mind, which isn't possible for every scenario.

Also, `Arrays` can only store **homogeneous** (similar) elements. If we try to add elements other than the one we initialized the array with, it will throw a [Compile Time Error](https://www.geeksforgeeks.org/java/types-of-errors-in-java-with-examples/).

> Example:

```java
Vehicle[] v = new Vehicle[200];

v[0] = new Vehicle();  // Valid
v[1] = new Truck();    // Error if Truck is NOT a subclass of Vehicle
```

This happens because the array is of type `Vehicle[]`, so it only accepts `Vehicle` objects or its **subclasses**.

#### **The Universal Type: `Object[]`**

We can partially overcome the type limitation using `Object[]`.

```java
Object[] arr = new Object[100];

arr[0] = new Vehicle();
arr[1] = new Truck();
arr[2] = "Valay";
arr[3] = 9999;
```

It works because the `Object` is the **_parent of all classes in Java_**, so it can store literally anything.

> [!important] Lack of Built-in Methods
> The main limitation of arrays is the lack of built-in methods for common operations like **Insert**, **Delete**, **Sort**, or **Update**. We must write the logic manually for every operation.

### Overcoming the Limitations

To overcome these hurdles, we use the **Collection Concept**.

> [!success] Why Collections?
>
> - **Growable Nature**: They can increase or decrease in size dynamically based on requirements, which is memory-efficient.
> - **Heterogeneous Storage**: They can store both homogeneous and heterogeneous objects without any issues.
> - **Rich API**: Built on standard data structures, providing ready-made methods so you don't have to write logic manually.

#### Difference between array and collection

| Feature              | **Arrays**                             | **Collection**                                         |
| -------------------- | -------------------------------------- | ------------------------------------------------------ |
| **Size**             | Fixed in Size                          | Growable in nature                                     |
| **Memory**           | Efficient - But not recommended        | Extra memory for resizing & metadata - But recommended |
| **Performance**      | Fast - But not recommended             | Slightly slower - Recommended                          |
| **Data Types**       | Supports Primitives + Objects          | Only Objects (use wrappers like `Integer`)             |
| **Type Nature**      | Only Homogeneous                       | Can be homogeneous or heterogeneous                    |
| **Built-in Methods** | None (manual coding)                   | Rich API (add, remove, sort, search)                   |
| **Best Use Case**    | Fixed-size, performance-critical tasks | Dynamic, real-world applications                       |

### Collection vs. Collection Framework

> [!info] Definitions
>
> - **Collection**: A way to represent a group of individual objects as a single entity.
> - **Collection Framework**: A unified architecture (set of classes and interfaces) for representing and manipulating collections.

In Java, **Collection is an interface**. It acts as a **rulebook** defining _what_ operations are possible (add, remove, access), while implementation classes like `ArrayList` or `LinkedList` define _how_ they are done.

All collection classes are classified into four main categories:

1. **List**
2. **Set**
3. **Queue**
4. **Map** (Part of the framework, but doesn't inherit from the `Collection` interface)

> [!tip] Collection (Interface) vs. Collections (Class)
>
> - **Collection (interface)** → defines the _rules_ (what to do).
> - **Collections (class)** → provides the _tools_ (utility methods like sorting/searching).

**Note**: All collection classes are available in the `java.util` package.

---

#### 1. List

- **Order**: Insertion order is preserved.
- **Duplicates**: Allowed.
- **Access**: By index.
- **Examples**: `ArrayList`, `LinkedList`, `Vector`, `Stack`.

#### 2. Set

- **Order**: No guarantee (usually).
- **Duplicates**: **NOT** allowed (strictly unique).
- **Best for**: When uniqueness is important (IDs, usernames).
- **Examples**: `HashSet`, `LinkedHashSet`, `TreeSet`.

#### 3. Queue

- **Order**: FIFO (First-In-First-Out).
- **Processing**: Elements added first are processed first.
- **Best for**: Task scheduling or buffering.
- **Examples**: `PriorityQueue`, `ArrayDeque`.

#### 4. Map

- **Structure**: Key-Value pairs.
- **Keys**: Must be unique.
- **Best for**: Fast lookups via a key.
- **Examples**: `HashMap`, `TreeMap`, `LinkedHashMap`.

## ⚡ Edge Case Analysis

### 1. Homogeneous vs. Heterogeneous

While Collections _can_ store heterogeneous objects (e.g., `list.add("String")` and `list.add(10)`), it is **strongly discouraged** in modern Java. Always use **Generics** (`List<String>`) to ensure type safety and avoid `ClassCastException` at runtime.

### 2. Primitives in Collections

Collections cannot store primitive types (int, char, double) directly. They only store **Objects**. When you add an `int` to an `ArrayList<Integer>`, Java perform **Autoboxing** to convert the `int` to an `Integer` object. This adds a slight performance overhead.

## 🙋 Interview Corner

**Q1: What is the difference between `Collection` and `Collections`?**

- **`Collection` (Interface):** The root interface for List, Set, and Queue.
- **`Collections` (Class):** A utility class containing static methods for operations like `sort()`, `shuffle()`, and `reverse()`.

**Q2: Why doesn't `Map` extend the `Collection` interface?**
A: Because `Collection` works with single elements, while `Map` works with **Key-Value pairs**. Their requirements and internal architectures are fundamentally different.

**Q3: What is the `Iterable` interface?**
A: It is the parent interface of `Collection`. It contains only one method: `iterator()`. Any class that implements `Iterable` can be used in a **for-each loop**.

**Q4: Which is better: Array or ArrayList?**
A: If the size is fixed and performance is ultra-critical (e.g., game engines), use **Array**. For dynamic real-world applications where data grows/shrinks, use **ArrayList**.
