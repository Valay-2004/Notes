# Exception Handling in Java

> [!abstract] What is an Exception?
> An **exception** is an **abnormal situation** that occurs during program execution and breaks the normal flow of the program.
>
> Real-life analogy :  
> You are driving a car on a smooth road (normal program flow). Suddenly a dog comes in front → you have two choices:
>
> 1. Crash (program terminates abnormally)
> 2. Apply brake and handle the situation (Exception Handling)
>
> Without handling → program crashes with ugly error.  
> With handling → program continues gracefully and shows a friendly message.

**Why do we need Exception Handling?**

- Prevent sudden program termination (crash).
- Give meaningful messages to the user.
- Recover from errors (e.g., ask user to re-enter input).
- Make code more **robust** and **professional**.
- Separate normal logic from error-handling logic.

## Exception Hierarchy (Must Know)

All exceptions come from **`Throwable`** class.

```text
Throwable (Parent)
├── Error (Serious JVM issues - OutOfMemoryError, StackOverflowError, etc.)
│   → Usually NOT handled by programmers
└── Exception (Recoverable issues - we handle these)
    ├── RuntimeException (Unchecked Exceptions)
    │   ├── ArithmeticException
    │   ├── NullPointerException
    │   ├── ArrayIndexOutOfBoundsException
    │   ├── ClassCastException
    │   └── ...
    └── Checked Exceptions (must be handled or declared)
        ├── IOException
        ├── SQLException
        ├── FileNotFoundException
        └── ...
```

> [!tip] Checked vs Unchecked (Very Important)
>
> - **Unchecked (RuntimeException)** → Happen at runtime. Compiler does **not** force you to handle them.  
>   Example: Dividing by zero, accessing null object.  
>   → You **should** still handle them for good code.
> - **Checked Exceptions** → Compiler **forces** you to handle or declare them (using try-catch or throws).  
>   Example: Reading a file that may not exist (`IOException`).  
>   → These are "expected" problems.

## 1. try-catch Block (Basic Handling)

```java
public class ExceptionDemo {
    public static void main(String[] args) {
        try {
            int a = 10;
            int b = 0;
            int result = a / b;           // This throws ArithmeticException
            System.out.println("Result: " + result);  // This line will NOT execute
        }
        catch (ArithmeticException e) {   // Specific catch
            System.out.println("Error: Cannot divide by zero!");
            System.out.println("Message: " + e.getMessage());
            // e.printStackTrace();       // Shows full error details (useful for debugging)
        }

        System.out.println("Program continues after handling exception...");
    }
}
```

**Line-by-line Explanation:**

- Code inside `try` is risky.
- If exception occurs → control immediately jumps to matching `catch` block.
- After catch → program continues (does not stop).

**Multiple catch blocks** (Catch specific first, then general):

```java
try {
    // risky code
} catch (ArithmeticException e) {
    System.out.println("Math error");
} catch (NullPointerException e) {
    System.out.println("Null pointer issue");
} catch (Exception e) {          // Parent catch - catches everything else
    System.out.println("Some other exception: " + e);
}
```

> [!warning] Rule:  
> Always put specific exceptions before the general `Exception` catch. Otherwise compile error.

## 2. finally Block (Cleanup Code)

`finally` block **always executes**, whether exception occurs or not.  
Best for cleanup (closing files, database connections, etc.).

```java
try {
    System.out.println("Trying to open file...");
    // code that may throw exception
} catch (Exception e) {
    System.out.println("Error occurred");
} finally {
    System.out.println("Finally block: Closing resources...");
    // This runs ALWAYS
}
```

## 3. throw vs throws

- **`throw`** → Used to **manually throw** an exception inside a method.
- **`throws`** → Used in **method signature** to declare that this method **may throw** a checked exception (passes responsibility to caller).

```java
// throw example (manual)
static void checkAge(int age) {
    if (age < 18) {
        throw new ArithmeticException("You are not eligible! Age must be 18+");
    }
    System.out.println("Welcome!");
}

// throws example (declaration)
static void readFile() throws IOException {
    // code that can throw checked exception
    FileReader fr = new FileReader("abc.txt");
}
```

## 4. Custom (User-Defined) Exception

Create your own exception class for business logic.

```java
// Custom Checked Exception
class MyException extends Exception {
    MyException(String msg) {
        super(msg);               // pass message to parent
    }
}

// Usage
class CustomDemo {
    static void validate(int age) throws MyException {
        if (age < 18) {
            throw new MyException("Custom Error: You are too young!");
        }
    }

    public static void main(String[] args) {
        try {
            validate(16);
        } catch (MyException e) {
            System.out.println(e.getMessage());
        }
    }
}
```

## 5. try-with-resources (Auto Resource Management)

Introduced in Java 7, this feature automatically closes resources (files, sockets, database connections) at the end of the `try` block. The resource must implement `java.lang.AutoCloseable`.

```java
import java.io.*;

public class TryWithResourcesDemo {
    public static void main(String[] args) {
        // Resource is declared inside parentheses
        try (BufferedReader br = new BufferedReader(new FileReader("test.txt"))) {
            String line = br.readLine();
            System.out.println(line);
        } catch (IOException e) {
            System.out.println("File error: " + e.getMessage());
        }
        // br is AUTOMATICALLY closed here - no finally block needed!
    }
}
```

> [!tip] Advantage
> It reduces "boilerplate" code and prevents memory leaks caused by forgetting to close resources in a `finally` block.

## Quick Summary Table

| Concept          | Purpose                             | Keyword           | Checked/Unchecked |
| ---------------- | ----------------------------------- | ----------------- | ----------------- |
| try-catch        | Handle exception gracefully         | try, catch        | Both              |
| finally          | Cleanup code (always runs)          | finally           | -                 |
| throw            | Manually throw exception            | throw             | Both              |
| throws           | Declare checked exception in method | throws            | Mostly Checked    |
| Custom Exception | Meaningful business error messages  | extends Exception | You decide        |

> [!success] Memory Trick
>
> - Exception = Unexpected guest at party.
> - try = "Let's try this..."
> - catch = "If guest causes problem, handle him politely."
> - finally = "Clean up the mess no matter what."
> - throw = "You! Get out right now!"
> - throws = "Not my responsibility — next person handle it."
> - Custom = Your own security guard with custom warning.

## ⚡ Edge Case Analysis

### 1. `finally` vs `return` (The Power Struggle)

**Scenario:** If both `try` and `finally` have a `return` statement, who wins?
**Result:** The `finally` block wins. The value returned by `finally` will override the one from `try` or `catch`.

```java
public int test() {
    try { return 1; }
    finally { return 2; } // This returns 2!
}
```

### 2. `System.exit(0)`

**Scenario:** Is there any way to skip the `finally` block?
**Result:** Yes. If you call `System.exit(0)` inside the `try` or `catch` block, the JVM shuts down, and `finally` will **not** execute.

### 3. Catching `Throwable`

**Scenario:** Should we use `catch (Throwable t)`?
**Result:** Generally **No**. This catches both `Exceptions` and `Errors` (like `OutOfMemoryError`). Handling system-level errors is usually impossible and dangerous.

## 🙋 Interview Corner

**Q1: Difference between `final`, `finally`, and `finalize`?**

- `final`: Keyword to make a variable/method/class constant/un-overridable/un-inheritable.
- `finally`: Block used with `try-catch` for cleanup code.
- `finalize`: (Deprecated) Method called by the Garbage Collector before an object is destroyed.

**Q2: Can we have a `try` block without a `catch` block?**
A: **Yes**, but only if there is a `finally` block or if you are using **try-with-resources**. A bare `try` block alone will cause a compile error.

**Q3: What is "Exception Propagation"?**
A: If an exception is not caught in the method where it occurs, it "drops" down the call stack to the previous method. If it's never caught, the program terminates. Note: Checked exceptions must be declared using `throws` to propagate.

**Q4: What happens if an exception is thrown inside a `catch` or `finally` block?**
A: It behaves like any other exception. If it's in `finally`, it will "swallow" any previous exception thrown in the `try` block and propagate upwards.

**Q5: Why is it bad to `catch (Exception e)` every time?**
A: It makes the code less precise. It might catch a `NullPointerException` (which is a bug) when you were only expecting a `FileNotFoundException` (which is a valid external error).
