# 🚀 Exception Handling Interview Questions

A collection of top interview questions for Java Exception Handling, categorized by difficulty.

---

## 1. Basic Questions (Level: Beginner)

**Q1: What is Exception Handling?**

- It is a mechanism to handle runtime errors so that the normal flow of the application can be maintained.

**Q2: What is the base class for all Exceptions and Errors?**

- **`java.lang.Throwable`** is the root class of the Java Exception hierarchy.

**Q3: Difference between `Checked` and `Unchecked` exceptions?**

- **Checked:** Checked at compile-time (e.g., `IOException`). You MUST handle or declare them.
- **Unchecked:** Checked at runtime (e.g., `NullPointerException`). You are not forced to handle them.

**Q4: Can we use `try` without `catch`?**

- Yes, provided there is a `finally` block or it is a `try-with-resources` statement.

**Q5: What is the purpose of the `finally` block?**

- It is used to execute important code such as closing resources (files, DB connections), regardless of whether an exception occurred or not.

---

## 2. Intermediate Questions (Level: Developer)

**Q1: Difference between `throw` and `throws`?**

- `throw`: Used to explicitly throw an exception from a method or block of code.
- `throws`: Used in the method signature to declare that the method might throw one or more exceptions.

**Q2: Can we catch multiple exceptions in a single catch block?**

- Yes (since Java 7), using the pipe `|` operator: `catch (ArithmeticException | NullPointerException e)`.

**Q3: What is "Exception Propagation"?**

- If an exception is not caught, it is thrown to the previous method in the call stack. This continues until it's caught or the stack is empty.

**Q4: What is `try-with-resources`?**

- A `try` statement that declares one or more resources. It ensures that each resource is closed at the end of the statement.

**Q5: Can we override a method that throws a checked exception?**

- Yes, but the overriding method cannot throw a "broader" checked exception than the parent method. It can throw the same, a narrower one, or none at all.

---

## 3. Expert Questions (Level: Senior)

**Q1: What happens if `return` is present in both `try` and `finally`?**

- The `return` in the `finally` block will execute and override the `return` in the `try` or `catch` block.

**Q2: What is "Suppressed Exceptions"?**

- When using `try-with-resources`, if an exception is thrown in the `try` block AND another while closing the resource, the first one is thrown, and the second is "suppressed" (available via `e.getSuppressed()`).

**Q3: Can we catch an `Error` in Java?**

- Technically yes, `catch (Error e)`, but it's strongly discouraged because Errors (like `OutOfMemoryError`) usually indicate fatal JVM states that shouldn't be handled.

**Q4: How does the JVM handle an uncaught exception?**

- It prints the stack trace and the exception message to the console and terminates the thread where the exception occurred.

**Q5: What is the "Custom Exception" best practice?**

- Always extend `Exception` (for checked) or `RuntimeException` (for unchecked) and provide a constructor that takes a `String message` to pass to the parent `super(message)`.

---

## 💡 Expert Tips

1. **Never swallow exceptions:** Don't leave a catch block empty. At least log the error.
2. **Specific over General:** Catch `FileNotFoundException` before `IOException`.
3. **Log or Throw, not both:** Don't log an exception and then re-throw it; it litters the logs with duplicate stack traces.

---

## One-liner to Remember

> **Exceptions are for exceptional cases — don't use them for normal control flow!**
