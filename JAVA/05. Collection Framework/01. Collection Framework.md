# Collection Framework

## What is a Collection and Why we need it?

> [!NOTE]
> What is a collection ?
> A `collection` is simply a group of objects, materials, or any other things that are in a group.

So, Similar to that `Collection Framework` can be said to be a group of `set of Classes` and `set of Interfaces`.

### BUT WHY DO WE NEED THIS ?

We know `Primitive Data Types` are used to store single value into a variable
For example:

- `int x = 10;`
- `int y = 25;`
- `int xy = 30;`

So if one wants to store 1000s of values, then we would need to declare that many number of variables in our program which is not feasible at all.

> I mean writing a thousand values and using them would be like a LOT OF WORK.

Thus, Here comes the concept of `Arrays` in which we can store many elements having same data type.

> In simple terms -- An `array` is a `data structure` that stores a fixed-size, sequential collection of elements of the same data type in contiguous memory locations.

The main advantage of is that we can store a huge number of elements with a single variable like:

```java
int[] arr = new int[5];
arr[0] = 1;
...
arr[4] = 5;
```

In the above code we've created an array with a length of `5` then we have initialized indexes with elements form 1 to 5.
([Arrays are 0 indexed](https://albertkoz.com/why-does-array-start-with-index-0-65ffc07cbce8))

### **Limitations of array:**

As Arrays are fixed in size we cannot `increase` or `decrease` the size of an array based on our requirements.

> Thus, to use an array we must have a fixed size in our mind on which we will be working but that's not possible for every scenario.

Also `Arrays` can only store `homogenous` (similar) elements i.e. if we try to add elements other than the one we initialized the array with it will throw a [Compile Time Error](https://www.geeksforgeeks.org/java/types-of-errors-in-java-with-examples/).

> Example:

```java
Vehicle[] v = new Vehicle[200];

v[0] = new Vehicle();  // This is valid
v[1] = new Truck();    // Invalid (If Truck is not a subclass of Vehicle)
```

This happens because the array is of type `Vehicle[]` so it only accepts `Vehicle[]` objects or its `subclasses` (if any)

We can overcome this limitation with the help of an `Universal type` → `Object[]`
Example:

```java
Object[] arr = new Object[100];

arr[0] = new Vehicle();
arr[1] = new Truck();
arr[2] = "Valay";
arr[3] = 9999;
```

It works because the `Object` is the **_parent of all classes in Java_**, so it can store literally anything.

Main limitation of arrays is that there are no built-in methods for doing operations on arrays
Operations like:

- Insert
- Delete
- Sort
- Update

So for every requirement we must **write logic manually** for operations.

---

### Overcoming the Limitations

**Thus to overcome the above limitation, we should go for the `Collection Concept`**
Why?

- Collection are growable in nature -- meaning based on our requirements we can increase or decrease their size which is recommended if we look from memory POV.
- Collection can hold different types of elements not just a single type of elements. -- meaning it can store homogenous & heterogenous objects _without any issues_.
- Collection classes are built on top of standard data structure, so they already provide ready-made methods.
  > Collections have built-in methods, so no need to write logic manually.

#### Difference between array and collection

|                      |               **Arrays**               |                     **Collection**                     |
| :------------------: | :------------------------------------: | :----------------------------------------------------: |
|       **Size**       |             Fixed in Size              |                   Growable in nature                   |
|      **Memory**      |    Efficient - But not recommended     | Extra memory for resizing & metadata - But recommended |
|   **Performance**    |      Faste - But not recommended       |             Slightly slower - recommended              |
|    **Data Types**    |      Supports Primitive + Objects      |       Only Objects (use wrapper like `Integer`)        |
|   **Type Nature**    |            Only Homogenours            |          Can be homogeneous or heterogeneous           |
| **Build-in Methods** |          None (manual coding)          |          Rich API (add, remove, sort, search)          |
|  **Best Use Case**   | Fixed-size, performance-critical tasks |            Dynamic, real-world applications            |

### Collection & Collection Framework

> [!NOTE]
> **Collection**: If we want to represent a group of objects as single entity then we should go for Collections.
> **Collection framework**: It defines several classes and interfaces to represent a group of Objects as single entity

In Java, **Collection is an interface** because it only _defines what operations are possible_, not how they are done.  
It acts like a **rulebook** that says things like: you should be able to add, remove, or access elements.

Different classes (like `ArrayList`, `LinkedList`, etc.) follow this rulebook and provide their own implementation.

All the collection classes are classified into three categories

- List
- Set
- Queue
- Map

---

On the other hand, **Collections is a class** because it actually _contains code_ (methods).  
It provides ready-made utility functions like sorting, searching, reversing, etc., which you can directly use.
**All the collection classes are available in “`java.util`” (utility) package.**

---

So in simple terms:

- **Collection (interface)** → defines _what should be done_
- **Collections (class)** → provides _ready-made tools to do it_

---

##### 1. List

- A **_List_** is used when you want to **store items in order** and access them by position (index).
- It allows **duplicates**, because sometimes the same value can appear multiple times (like marks, names, etc.).
- It exists `To handle data where order matters` and we may need to access elements using and index.
- List is an Interface whose object can not be created directly.
- To work with this category we have to use following implementations class of list interface.
  - Ex: `ArrayList`, `Linked list`, `Vector`, `Stack`

##### 2. Set

- A **Set** is used when you want to store only **_unique_ values**.
- It does **Not allow Duplicates**, so if we try to add duplicate values it will be ignored.
- It exists `To handle where UNIQUENESS is important` (like IDs, usernames).
- Set is an interface whose object cannot be created directly.
- To work with this category, we have to use following implementations class of Set interface.
  - Ex: `HashSet`, `LinkedHashSet` and `TreeSet`

##### 3. Queue

- A **Queue** is used when data should be processed in a **specific order (FIFO)** → first in, first out (think of a queue on a counter).
- The element added first is removed first, just like a real-life queue.
- It exists `To manage tasks where order of processing matters,` like scheduling or buffering.
- It is an ordered list of objects
  - Ex: `PriorityQueue`, `BlockingQueue`

##### 4. Map

- **Map** is used to **store** data in a **key-value pairs**.
- Each key is unique, and it maps to a value (like a dictionary).
- It exists `To quickly find data using a key` instead of searching through everything.
- Map is an interface whose object cannot be created directly.
- To work with this category, we have to use following implementation classes of Map interface
  - Ex: `HashMap`, `LinkedHashMap`, `TreeMap`, `Hashtable`
